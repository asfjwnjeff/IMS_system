# IMS 保险管理系统 — 业务流程调研文档

> 日期：2026-06-30
> 版本：v1.1
> 状态：调研完成，待开发

---

## 一、角色与职责

| 角色 | 职责 |
|------|------|
| 客服 / 货代 | 从 COS/FMS 发起投保；填写/编辑投保单；发起批改；发起报案理赔；下载保单/缴费通知书/发票；BMS 发起付款 |
| 保险团队（审核员） | 第一级投保审核；投保最终确认；线下向保险公司提交投保/批改/理赔申请；回填保单/批改/理赔信息 |
| 主管 | 第二级投保审核；第二级批改审核；报案理赔审核 |
| 保险公司 | 出具正式保单/批单；开展理赔评估并反馈结果 |
| 钉钉 | 所有审批实例的执行平台（**原型阶段仅模拟审批流转**） |

---

## 二、核心流程全景

```
                          ┌─ 不需要批改 ──────────────────────────┐
                          │                                        │
COS/FMS发起 → 投保出单 → 投保完成 → 判断 ──→ 缴费取单 → 判断 ──→ 结束
                          │                 │             │
                          └─ 需要批改 ──→ 批单批改 ──→ ┘    │
                                                             │
                                          ┌─ 不发起理赔 ────┘
                                          │
                                          └─ 发起理赔 → 报案理赔 → 流程结束
```

## 三、投保出单阶段（流程起点）

### 3.1 流程图

```
COS/FMS 发起
    │
    ▼
① 填写/编辑投保单                      状态：待发起
    │                                  更多操作：保险费率选择 | 编辑 | 删除 | 发起审批
    ▼
② 选择保险费率（更多操作 → 保险费率选择）
    ├─ 保险公司名称（下拉单选）
    ├─ 单独报价（下拉：是/否）
    ├─ 费率（与保险公司双向联动）          ← 联动规则见 7.3
    └─ 预计保费（系统自动计算）
    │
    ▼
③ 发起审批                             状态：待发起 → 待审核（钉钉创建审批实例）
    │
    ▼
④ 保险团队审核 ──── 拒绝 ──→ 状态：审批拒绝                              
    │                          ↓ 用户直接编辑投保单，然后重新发起审批
    │ 通过                      
    ▼                          
⑤ 主管审核 ──── 拒绝 ──→ 状态：审批拒绝                                  
    │                          ↓ 同上
    │ 通过                      
    ▼                          
⑥ 保险团队投保最终确认 ──── 不确认 ──→ 退回保险团队重新审核（仍为待审核）
    │
    │ 确认 → 状态：审批通过 = 投保完成
    ▼
⑦ 保险团队线下向保险公司正式提交投保申请
    │
    ▼
⑧ 保险公司出具正式保单
    │
    ▼
⑨ 保险团队回填保单信息到系统（更多操作 → 保单信息回填）
    ↓                                  准入条件：状态 = 审批通过
✅ 投保出单完成
```

### 3.2 撤回与编辑规则

| 场景 | 行为 |
|------|------|
| 投保证在审批中（待审核） | 可「撤销审核」→ 状态回到"待发起"，**通知所有已审批过的人** |
| 审批被拒绝（审批拒绝） | 用户**直接在原记录上编辑**修改，然后重新「发起审批」 |
| 投保已完成（审批通过） | **不可撤回**，如需修改 → 必须走批单批改流程 |

## 四、批单批改阶段（可选环节）

### 4.1 触发条件

**必须**投保状态 = 审批通过之后，才可发起批改。在此之前如需修改 → 直接编辑或撤回。

### 4.2 批改单创建逻辑

```
点击「发起批改」
    │
    ▼
系统自动复制原投保单全部数据
    ├─ 新 ID（独立记录）
    ├─ 业务参考号 不变
    ├─ 申请类型 → "批改申请"
    ├─ 版本号 +1（如 V1 → V2）
    ├─ previousId → 原记录 ID
    ├─ isLatest → true（同时原记录 isLatest → false）
    └─ 状态 → 待发起
```

### 4.3 流程图

```
投保完成
    │
    ├── 不需要批改 ──→ 进入「缴费取单」
    │
    └── 需要批改
          │
          ▼
        ① 点击「发起批改」→ 系统自动复制生成批改单 → 用户修改需变更字段
          │
          ▼
        ② 发起审批 → 保险团队审核 ──拒绝──→ 用户直接编辑后重新发起
          │ 通过
          ▼
        ③ 主管审核 ──拒绝──→ 用户直接编辑后重新发起
          │ 通过
          ▼
        ④ 保险团队批改最终确认 ──不确认──→ 退回保险团队重新审核
          │ 确认
          ▼
        ⑤ 线下向保险公司提交批单申请 → 保险公司出具新保单
          │
          ▼
        ⑥ 回填批改信息到系统（更多操作 → 批改信息回填）
          ↓
✅ 批单批改完成 → 进入缴费取单
```

### 4.4 编辑页复用

批改单编辑页**复用投保单编辑页**，包含**全部字段**（不区分是否变更），用户自行修改需要批改的字段。

## 七、审核状态与状态机

### 7.1 状态定义

| 状态 | 含义 | 可用的更多操作 |
|------|------|---------------|
| `待发起` | 投保单草稿，尚未提交审批 | 保险费率选择、编辑、删除、发起审批 |
| `待审核` | 已提交，当前处于某一审核节点 | 撤销审核 |
| `审批通过` | 三级审批全部通过（投保完成） | 保单信息回填、批改信息回填、缴费账单、发票、发起批改 |
| `审批拒绝` | 任一级别审批不通过 | 编辑、发起审批 |

### 7.2 状态流转图

```
                    ┌──────────────────┐
                    │    待发起         │
                    └──────┬───────────┘
                           │ 发起审批
                           ▼
                    ┌──────────────────┐
            ┌───────│    待审核         │───────┐
            │ 拒绝   └──────────────────┘  通过  │
            ▼                                    ▼
     ┌──────────────┐                  ┌──────────────────┐
     │  审批拒绝      │                  │   审批通过         │
     │  (可编辑)      │                  │  (投保完成)        │
     └──────┬───────┘                  └────────┬─────────┘
            │ 发起审批                          │ 发起批改
            └─────────────┐                    ▼
                          │            ┌──────────────────┐
                          └───────────→│    待发起         │
                                       │  (新版本:批改申请) │
                                       └──────────────────┘
```

### 7.3 审批节点（当前版本不细分展示）

```
保险团队审核 → 主管审核 → 投保确认中 → 投保完成
      ←─── 均映射为 "待审核" 状态 ───→           ↑
                                               审批通过
```

> 后续版本在"待审核"下按节点细分展示。

## 八、版本管理方案

### 8.1 核心问题

批改后会产生同一 `businessRefNo` 的多条记录（如新增投保 V1 + 批改申请 V2），全部平铺在列表中会造成干扰。

### 8.2 方案：单列表 + Tab 聚合

```
┌──────────────────────────────────────────────┐
│ 投保申请表                   [● 当前有效] [○ 全部记录] │  ← Segmented 切换
│                                              │
│ ┌─────────────────────────────────────────┐  │
│ │ 业务参考号        │ 申请类型  │ 状态    │  │
│ ├─────────────────────────────────────────┤  │
│ │ PVG-AI26060171   │ 批改申请  │ 审批通过│  │  ← 正常显示
│ │ PAC-INTEL...     │ 新增投保  │ 审批通过│  │
│ └─────────────────────────────────────────┘  │
│                                              │
│ 切换到"全部记录"后：                           │
│                                              │
│ ┌─────────────────────────────────────────┐  │
│ │ 业务参考号        │ 申请类型  │ 状态    │  │
│ ├─────────────────────────────────────────┤  │
│ │ PVG-AI26060171   │ 批改申请  │ 审批通过│  │  ← 最新版本（正常色）
│ │ PVG-AI26060171   │ 新增投保  │ 审批通过│  │  ← 历史版本（灰色 + Tag「历史版本」）
│ │ PAC-INTEL...     │ 新增投保  │ 审批通过│  │
│ └─────────────────────────────────────────┘  │
│                                              │
│ 历史版本行处理：                               │
│  ● 文字颜色变灰                                │
│  ● 「历史版本」Tag                             │
│  ● 操作列仅保留「详情查看」，不可编辑/审批        │
└──────────────────────────────────────────────┘
```

### 8.3 数据模型扩展

`InsuranceApplication` 新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `isLatest` | `boolean` | 是否为该 businessRefNo 下的最新版本 |
| `version` | `number` | 版本号（1, 2, 3...） |
| `previousId` | `number \| null` | 关联的上一个版本 ID |

### 8.4 批改发起时的数据操作

```typescript
function createCorrection(original: InsuranceApplication): InsuranceApplication {
  return {
    ...original,
    id: generateNewId(),
    applicationType: '批改申请',
    version: original.version + 1,
    previousId: original.id,
    isLatest: true,           // 新记录成为最新版本
    approvalStatus: '待发起',
    applicationNo: generateNewApplicationNo(),
    insuranceCorrectionStatus: '',
    correctionActualPremium: 0,
  };
}
// 同时：original.isLatest = false
// 同时：original 作为历史版本快照保留
```

## 十、费率和保费计算规则

### 10.1 数据来源

| 数据 | 来源 | 维护入口 |
|------|------|----------|
| 汇率 | 后台「保费汇率配置」 | `/base/premiumexchangerateallocation` |
| 费率 | 后台「保费费率配置」 | `/base/detailsoftheinsurancerateconfiguration` |
| 预计保险金额 | 投保单中用户填写 | 编辑页「保险信息」卡片 |

### 10.2 计算逻辑

```
预计保费 = 汇率 × 费率 × 预计保险金额
```

### 10.3 保险费率选择弹窗交互

| 字段 | 组件 | 联动规则 |
|------|------|----------|
| 保险公司名称 | 下拉单选 | — |
| 单独报价 | 下拉：是/否 | — |
| 费率 | 下拉单选 | ① 选保险公司后，费率列表只展示该公司的费率；② 选费率后，自动带出对应保险公司 |
| 预计保费 | 只读 / 自动计算 | 汇率 × 费率 × 预计保险金额 |
