## 1. 查询功能自定义

### 1.1 概要

查询功能自定义允许用户在投保申请列表页自由选择显示哪些搜索字段（11 选 N），配置通过 localStorage + API 双写持久化，每个用户的搜索偏好跨会话保留。搜索采用**实时过滤**机制，输入即过滤，无需点击搜索按钮。

### 1.2 搜索字段定义（11 个字段）

搜索字段定义位于 `searchFieldDefs` 数组，每个字段包含 key、label、type、options、width 属性。

| 序号 | key | 字段名称 | 类型 | 宽度(px) | 选项 |
|------|-----|----------|------|----------|------|
| 1 | `policyNo` | 投保单号 | input | 150 | -- |
| 2 | `jobidref` | 业务参考号 | input | 150 | -- |
| 3 | `applicationTime` | 投保时间 | dateRange | 240 | -- |
| 4 | `workflowStatus` | 审批状态 | select | 130 | Draft, WaitStart, Approving, Approved, Reject |
| 5 | `insuranceCompanyPolicyStatus` | 保险公司保单状态 | select | 150 | 已承保, 已批改, 待承保 |
| 6 | `policyNumber` | 保单单号 | input | 150 | -- |
| 7 | `policyHolderNameDesc` | 投保人企业名称 | input | 180 | -- |
| 8 | `insuredCompanyName` | 被保人企业名称 | input | 180 | -- |
| 9 | `productNameCn` | 中文商品名称 | input | 180 | -- |
| 10 | `insuranceCompanyName` | 保险公司名称 | input | 150 | -- |
| 11 | `dataSources` | 数据来源 | select | 100 | fms, cos |

**字段类型说明：**

- `input`：普通文本输入框，模糊匹配。
- `select`：下拉选择，精确匹配。
- `dateRange`：日期范围（当前版本 UI 上尚未实现日期范围选择器，仅定义）。

### 1.3 默认可见字段

初始状态仅 **4 个字段** 默认可见，其余 7 个隐藏：

| 默认顺序 | key | 默认可见 | 说明 |
|----------|-----|----------|------|
| 0 | `policyNo` | **是** | 投保单号 |
| 1 | `jobidref` | **是** | 业务参考号 |
| 2 | `applicationTime` | **是** | 投保时间 |
| 3 | `workflowStatus` | 否 | 审批状态 |
| 4 | `insuranceCompanyPolicyStatus` | 否 | 保险公司保单状态 |
| 5 | `policyNo` | 否 | 投保单号（重复，疑为代码 Bug） |
| 6 | `policyHolderNameDesc` | 否 | 投保人企业名称 |
| 7 | `insuredCompanyName` | **是** | 被保人企业名称 |
| 8 | `productNameCn` | 否 | 中文商品名称 |
| 9 | `insuranceCompanyName` | 否 | 保险公司名称 |
| 10 | `dataSources` | 否 | 数据来源 |

> **注意**：`policyNo` 在默认配置中出现两次（order 0 和 order 5），第二次的 `visible: false` 实际不会生效，因为 key 重复，toggle 时会命中第一个。

**关键交互逻辑：**

- 每个字段显示为一个可点击标签，按 `order` 排序。
- 已选中字段：蓝底蓝边框 + "✓" 前缀。
- 未选中字段：白底灰边框 + "+" 前缀。
- 点击标签调用 `searchConfig.toggle(key)` 切换可见性。
- 点击"重置默认"调用 `searchConfig.reset()` 恢复初始状态。
- 点击"确定"仅关闭面板（`setSearchPanelOpen(false)`），不触发额外保存。

### 1.5 持久化机制（localStorage + API 双写）

搜索字段配置使用 `usePersistedConfig` hook，key 为 `ims_search_fields_v2`。

```
持久化流程：

  toggle / reorder / reset
         │
         ▼
    save(newConfig)
         │
         ├──► localStorage.setItem("ims_ims_search_fields_v2", JSON)
         │    (同步写入，即时生效)
         │
         └──► POST /api/column-configs
              { configKey: "ims_search_fields_v2", configData: [...] }
              (异步写入，静默失败)
```

**加载优先级：**

```
1. GET /api/column-configs?key=ims_search_fields_v2  ← 优先（服务端用户配置）
   │  失败
   ▼
2. localStorage.getItem("ims_ims_search_fields_v2")   ← 降级（本地缓存）
   │  缓存 key 数量 ≠ 默认值数量 → 丢弃缓存，使用默认值
   ▼
3. defaultSearchFields                                ← 最终兜底
```

**缓存失效策略：** 当缓存的字段数量与当前 `defaultSearchFields` 数量不一致时（表示配置结构发生变更），自动丢弃旧缓存并使用默认值。

### 1.6 搜索过滤逻辑

采用**实时过滤（Live Filter）**机制，通过 `useMemo` 实现：

```typescript
const filtered = useMemo(() => {
  return applications.filter((a) => {
    // 1. 视图筛选：当前有效模式仅显示 isLatest === true 的记录
    if (viewMode === 'latest' && !a.isLatest) return false;

    // 2. 多条件 AND 模糊匹配
    for (const [key, val] of Object.entries(searchValues)) {
      if (!val) continue;
      const v = String(val).toLowerCase();
      const fieldVal = getFieldString(a, key);
      if (!fieldVal.toLowerCase().includes(v)) return false;
    }

    return true;
  });
}, [applications, viewMode, searchValues]);
```

**核心特性：**

- **AND 逻辑**：多个搜索条件之间为"与"的关系。
- **模糊匹配**：所有 input 类型字段使用 `toLowerCase().includes()` 进行不区分大小写的子串匹配。
- **实时响应**：搜索条件变化立即重新计算，无需点击"搜索"按钮。搜索按钮在当前实现中为占位（注释 `/* search already live-filtered */`）。
- **重置按钮**：点击"重置"调用 `setSearchValues({})` 清空所有搜索条件。

### 1.7 视图切换（当前有效 / 全部记录）

在查询卡片顶部以分段控件（Segmented Control）形式呈现：

| 视图模式 | `viewMode` 值 | 过滤条件 | 用途 |
|----------|---------------|----------|------|
| 当前有效 | `'latest'` | `a.isLatest === true` | 仅显示最新版本记录 |
| 全部记录 | `'all'` | 无额外过滤 | 显示所有版本（含历史版本） |

历史版本行在表格中以 `opacity-50`（半透明）样式呈现。

### 2.4 每个操作的代码逻辑

#### 2.4.1 勾选显隐（Toggle）

```typescript
// usePersistedConfig.ts
const toggle = useCallback(
  (key: string) => {
    const newConfig = config.map((item) =>
      item.key === key ? { ...item, visible: !item.visible } : item
    ) as T;
    save(newConfig);  // 触发 localStorage + API 双写
  },
  [config, save]
);
```

采用不可变模式：通过 `map` 创建新数组，仅修改匹配项。

#### 2.4.2 拖拽排序（Drag & Drop）

基于 HTML5 原生 Drag & Drop API：

1. **dragStart**：记录拖拽起始索引 `dragIndex`。
2. **dragOver**：记录悬停目标索引 `dragOverIndex`（用于蓝色上边框提示线）。
3. **drop**：调用 `onReorder(dragIndex, dropIndex)`。
4. **dragEnd**：清除所有拖拽状态。

拖拽中的视觉反馈：
- 被拖拽项：`opacity-40` + 蓝色背景。
- 悬停目标：顶部 2px 蓝色边框（`border-t-2 border-[var(--accent)]`）。

```typescript
// usePersistedConfig.ts
const reorder = useCallback(
  (from: number, to: number) => {
    const newConfig = [...config] as T;    // 浅拷贝
    const [removed] = newConfig.splice(from, 1);  // 移除原位置
    newConfig.splice(to, 0, removed);             // 插入目标位置
    save(newConfig);
  },
  [config, save]
);
```

#### 2.4.3 上移 / 下移

```typescript
// ColumnSettings.tsx
const handleMoveUp = useCallback((index: number) => {
  if (index <= 0) return;               // 边界保护
  onReorder(index, index - 1);          // 与上一个元素交换
}, [onReorder]);

const handleMoveDown = useCallback((index: number) => {
  if (index >= columns.length - 1) return;  // 边界保护
  onReorder(index, index + 1);              // 与下一个元素交换
}, [columns.length, onReorder]);
```

首项禁用 ↑，末项禁用 ↓（`disabled:opacity-25 disabled:cursor-not-allowed`）。

#### 2.4.4 置顶

```typescript
// usePersistedConfig.ts
const moveToTop = useCallback(
  (key: string) => {
    const idx = config.findIndex((item) => item.key === key);
    if (idx <= 0) return;    // 已在顶部则无操作
    reorder(idx, 0);         // 复用 reorder 逻辑
  },
  [config, reorder]
);
```

#### 2.4.5 恢复默认

```typescript
// usePersistedConfig.ts
const reset = useCallback(() => {
  save(defaults);   // 直接写入初始配置
}, [defaults, save]);
```

### 2.5 列配置持久化

与搜索字段配置使用同一 `usePersistedConfig` hook，配置 key 为 `ims_table_columns_v2`。

**数据结构：**

```typescript
// 存储格式：T[]
[
  { key: "id", label: "序号", visible: true, order: 0 },
  { key: "jobidref", label: "业务参考号", visible: true, order: 1 },
  // ... 共 33 项
]
```

**注意**：表格实际渲染时，`order` 字段仅用于 ColumnSettings 面板内的排序展示。表格列的渲染顺序取决于 `columnConfig.config` 数组本身的顺序（由 `reorder` 操作维护）。

存储策略与搜索字段配置完全一致（见 1.5 节）：API 优先 → localStorage 降级 → 默认值兜底，写操作采用 localStorage 同步 + API 异步的双写模式。

### 2.6 操作列固定（Sticky）机制

操作列作为表格最后一列，通过 CSS sticky + box-shadow 实现固定效果：

```html
<!-- 表头 -->
<TableHead className="
  w-[80px]          宽 80px
  text-center       居中
  sticky right-0    固定于右侧
  z-20              高于普通列
  bg-[var(--bg-subtle)]  背景色覆盖
  shadow-[-4px_0_8px_rgba(0,0,0,0.06)]  左侧阴影
">
  操作
</TableHead>

<!-- 单元格 -->
<TableCell className="
  sticky right-0    固定于右侧
  z-10              z-index 低于表头
  bg-[var(--bg-surface)]  背景色覆盖
  shadow-[-4px_0_8px_rgba(0,0,0,0.06)]  左侧阴影
">
  ...操作按钮
</TableCell>
```

**关键实现细节：**

- 操作列不参与列设置（不在 `allColumns` / `visibleColumns` 中）。
- 始终渲染在表格最右侧。
- 横向滚动时操作列保持可见。
- 通过 `z-20`（表头）和 `z-10`（单元格）确保不被其他列覆盖。

## 3. 显示变更（版本对比）

### 3.1 概要

"显示变更"功能用于对比当前版本与最近一次历史版本之间的字段差异，以黄色高亮标注变更字段并显示原值。支持详情页（只读查看）和编辑页（编辑对比）两种场景。

### 3.2 触发条件

"显示变更"按钮的显示条件：

```typescript
// 详情页 & 编辑页
const compareVersion: HistoryVersion | null =
  versions.length > 0 ? versions[versions.length - 1] : null;

// 仅当存在至少一个历史版本时才渲染按钮
{compareVersion && (
  <Button onClick={() => setShowChanges(!showChanges)}>
    {showChanges ? '隐藏变更' : '显示变更'}
    {!showChanges && changedCount > 0 && `（${changedCount}）`}
  </Button>
)}
```

**条件总结：**

| 条件 | 详情页 | 编辑页 |
|------|--------|--------|
| 需要有历史版本 | 是 | 是 |
| 需要可编辑（canEdit） | 否 | 否（编辑页始终显示如果存在历史版本） |
| compareVersion 非 null | 是 | 是 |

数据来源：`GET /api/history?type=versions&applicationId=${id}`，取返回数组的最后一个元素作为对比基准（`versions[versions.length - 1]`）。

### 3.3 变更计数的计算逻辑

```typescript
// 详情页：遍历所有 7 个段落的全部字段
const changedCount = compareVersion
  ? sections.reduce(
      (count, s) => count + s.fields.filter((f) => isFieldChanged(f.key)).length,
      0
    )
  : 0;

// 编辑页：仅遍历 5 个可编辑段落（排除保单回填信息和批改信息）
const editSections = sections.filter(
  (s) => s.title !== '保单回填信息' && s.title !== '批改信息'
);
const changedCount = compareVersion
  ? editSections.reduce(
      (c, s) => c + s.fields.filter((f) => isChanged(getSectionKey(f.key), f.key)).length,
      0
    )
  : 0;
```

**编辑页排除两个段落的原因：** 保单回填信息和批改信息为审批通过后才填写的字段，在初始编辑阶段无历史值可对比。

### 3.4 单个字段变更判断逻辑

核心方法是 `JSON.stringify` 序列化对比：

```typescript
// 详情页
const isFieldChanged = (key: string): boolean => {
  if (!showChanges || !compareVersion) return false;
  let prev: unknown, curr: unknown;
  try {
    const prevData = typeof compareVersion.data === 'string'
      ? JSON.parse(compareVersion.data)
      : compareVersion.data;
    prev = (prevData as Record<string, unknown>)[key];     // 历史版本中的值
    curr = getFieldValue(app!, key);                       // 当前版本中的值
  } catch { return false; }
  return JSON.stringify(prev) !== JSON.stringify(curr);    // 序列化对比
};
```

```typescript
// 编辑页（参数略有不同）
const isChanged = (sk: string, key: string): boolean => {
  if (!showChanges || !compareVersion) return false;
  try {
    const data = typeof compareVersion.data === 'string'
      ? JSON.parse(compareVersion.data)
      : compareVersion.data;
    const prev = (data as Record<string, unknown>)[key];
    const curr = getFieldValue(sk, key);                    // 从编辑表单中读值
    return JSON.stringify(prev) !== JSON.stringify(curr);
  } catch { return false; }
};
```

**对比逻辑说明：**

- 使用 `JSON.stringify` 进行深度值对比，可正确处理 `null`、`undefined`、数字、字符串、数组等不同类型的差异。
- 包括 `undefined -> 'some value'` 或 `'value' -> undefined` 的变更。
- 当 `showChanges === false`（未点击"显示变更"按钮）时，所有字段返回 `false`（不高亮）。

### 3.7 历史版本数据来源

历史版本通过 `GET /api/history?type=versions&applicationId=${id}` 接口获取，返回数据结构：

```typescript
interface HistoryVersion {
  id: string;
  applicationId: string;
  version: number;       // 版本号（如 1, 2, 3）
  timestamp: string;     // 创建时间
  label: string;         // 版本标签（如 "2026-06-15 编辑"、"原始版本"）
  data: string;          // JSON 字符串，存储该版本的完整投保申请数据
}
```

对比时：
- 基准版本：`versions[versions.length - 1]`（最后一个元素，即最近一次历史版本，version 号最大）。
- 当前版本：内存中的 `app` 对象（列表页数据）或 `editValues` + `rootValues`（编辑页表单数据）。

详情页在组件挂载时加载历史版本（`useEffect`），存储在 `versions` 状态中。`HistoryPanel` 组件另一次独立加载同一数据源用于"历史版本"Tab 展示。

### 3.8 隐藏变更按钮的行为

点击"显示变更"按钮后：

1. `showChanges` 切换为 `true`。
2. 按钮文案变为"隐藏变更"，按钮样式变为黄色（`bg-yellow-500 hover:bg-yellow-600`）。
3. 变更提示条出现。
4. 所有 `isFieldChanged()` 返回实际对比结果，黄色高亮生效。
5. 按钮上不再显示变更计数。

再次点击：
1. `showChanges` 切换为 `false`。
2. 按钮恢复 `outline` 样式，文案恢复"显示变更"。
3. 变更提示条消失。
4. 所有字段恢复正常样式（`isFieldChanged()` 全部返回 `false`）。
5. 按钮上重新显示变更计数（如"（5）"）。

### 3.9 详情页 vs 编辑页的差异

| 对比维度 | 详情页 | 编辑页 |
|----------|--------|--------|
| 文件路径 | `applications/[id]/page.tsx` | `applications/[id]/edit/page.tsx` |
| 对比段落数 | 7 段（全部） | 5 段（排除保单回填、批改信息） |
| 对比数据源 | 只读 `app` 对象 + `getFieldValue()` | 编辑表单 `rootValues` + `editValues[section]` |
| 高亮样式 | `bg-yellow-50 border-2 p-1.5` | `bg-yellow-50 border-2 -m-1.5 p-1.5` |
| 原值显示 | `getOldValue(key)` → `formatVal()` | `getPrevVal(sk, key)` → 空值显示"（空）" |
| 提示条 | 含版本标签和说明文字 | 简化版，仅含版本号和变更数 |
| 字段值格式化 | `formatVal()` 函数（数字千分位、状态映射） | `String()` 直接转换 |
| 保存时的脏检测 | 无（只读） | 有 `dirty` 状态，离开时弹窗确认 |
| 离开保护 | 无 | `dirty && confirm('有未保存的修改，确定离开？')` |

### 3.10 数据流汇总

```
页面加载
  │
  ├──► GET /api/history?type=versions&applicationId={id}
  │    返回 [{ version: 1, data: "{...}" }, { version: 2, data: "{...}" }]
  │
  ▼
versions state = [...]
  │
  ▼
compareVersion = versions[versions.length - 1]  // V2, 最近一次历史版本
  │
  ▼
用户点击 [显示变更]
  │
  ▼
showChanges = true
  │
  ▼
isFieldChanged(key):
  ├── JSON.parse(compareVersion.data)[key]   ← 历史值
  ├── getFieldValue(app, key) / editValues   ← 当前值
  └── JSON.stringify(prev) !== JSON.stringify(curr)  → true/false
  │
  ▼
changed = true → 黄色高亮 + 显示原值
changed = false → 正常样式
```
